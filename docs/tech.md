# Техническая спецификация

## Назначение

Проект представляет собой локальное офлайн-приложение для работы с архивом сообщений VK:
- импорт сообщений из HTML-архива;
- хранение в SQLite;
- полнотекстовый поиск;
- просмотр переписок через web UI;
- экспорт сообщений.

Пример входных данных лежит в `./messages`.

## Архитектура

Проект состоит из нескольких частей:
- `backend/`
  - FastAPI API
  - импорт архива
  - SQLite + SQLAlchemy 2.0
  - Alembic
  - structlog
- `web/`
  - исходники frontend
- `web-out/`
  - собранный frontend
  - раздаётся backend как статический SPA
- `deploy/migrations/`
  - миграции Alembic

## Запуск

Сценарий локального запуска:
1. Запустить backend.
2. Backend читает TOML-конфиг из `VK_ARCHIVE_CONFIG` или из аргумента `--config`.
3. Backend проверяет sqlite-файл.
4. Если файл базы отсутствует, выполняется `alembic upgrade head`.
5. Backend поднимает API и раздаёт frontend из `web-out`.

Frontend не требуется запускать отдельным процессом для обычного использования собранного приложения.

## Основные допущения

1. Приложение работает локально, без доступа к интернету.
2. Полный импорт обычно делается один раз.
3. Повторный импорт допустим и должен быть идемпотентным.
4. Уникальность сообщения задаётся парой:
   - `user_id`
   - `timestamp` с точностью до секунды
5. Владелец архива считается пользователем с `user_id = 0`.
6. На старте поддерживается формат архива, аналогичный данным из `./messages`.
7. Парсер должен быть готов к поддержке нескольких близких вариантов формата.

## Backend

### Слои

Backend разделён как минимум на следующие слои:
- `api`
- `services`
- `repositories`
- `parsers`
- `models`
- `db`
- `core`

### Логирование

Используется `structlog` поверх `logging`.

Логи:
- пишутся в stdout;
- пишутся в файл;
- пишутся в JSON;
- сохраняют Unicode в читаемом виде;
- используют простую файловую ротацию.

Настройки:

```toml
[logging]
level = "INFO"
path = "logs/backend.log"
max_bytes = 1048576
backup_count = 3
```

### Импорт

Импорт работает по файлам `messages*.html`, а не по всему архиву одним большим блоком.

Это нужно для:
- понятного прогресса;
- более дешёвого bulk insert;
- меньшего потребления памяти;
- удобной диагностики ошибок.

Во время импорта логируются:
- старт импорта;
- успешная обработка каждого файла;
- ошибка по конкретному файлу с причиной;
- итог импорта.

### Раздача frontend

Backend раздаёт:
- `/api/v1/*` — API;
- `/assets/*` — статические assets из `web-out/assets`;
- `/` — `web-out/index.html`;
- остальные web-маршруты — через SPA fallback на `index.html`.

## База данных

### Технологии

- `sqlite`
- `SQLAlchemy 2.0`
- `alembic`
- `FTS5`

### `conversations`

Одна запись на одного собеседника.

Поля:
- `user_id` — primary key
- `display_name` — последнее известное имя
- `first_message_at`
- `last_message_at`
- `message_count`
- `created_at`
- `updated_at`

### `messages`

Поля:
- `id`
- `user_id`
- `timestamp`
- `direction`
- `text`
- `has_attachments`
- `created_at`

Примечания:
- отдельный `conversation_id` не нужен;
- уникальность задаётся по `(user_id, timestamp)`;
- служебные сообщения отдельно не сохраняются;
- полезный текст вложений добавляется в `text`.

### Индексы

Минимальные индексы:
- уникальный индекс на `(user_id, timestamp)`
- индекс на `(user_id, timestamp)`
- индекс на `timestamp`

### Полнотекстовый поиск

Используется SQLite `FTS5`.

Поиск идёт только по тексту сообщений.

Поддерживаются два режима:
- `simple`
  - обычный текст пользователя
  - backend сам строит безопасный FTS-запрос
- `fts`
  - пользователь сам задаёт FTS-выражение
  - доступны `AND`, `OR`, `NOT`, фразы, префиксный поиск

## Парсер архива

### Что извлекается

Из HTML-файла извлекаются:
- `user_id`
- `display_name`
- `owner_user_id`
- сообщения
- время сообщения
- направление
- вложения в текстовом виде

### Особенности

Парсер поддерживает:
- кириллицу в `cp1251`;
- сообщения с вложениями;
- время с одно- и двузначным часом, например `0:40:56` и `10:22:53`.

## API

Префикс:
- `/api/v1`

Все timestamps в API — ISO 8601 UTC.

### `POST /api/v1/add`

Импорт архива из локального пути.

Request:

```json
{
  "path": "/absolute/or/relative/path/to/messages"
}
```

Response:
- `text/event-stream`

Пример событий:

```text
event: progress
data: {"total":3,"remains":2,"errors":0}

event: done
data: {"total":3,"remains":0,"errors":0,"imported":97,"skipped":0}
```

Семантика:
- `total` — число файлов
- `remains` — сколько файлов осталось
- `errors` — число файлов с ошибкой
- `imported` — сколько сообщений импортировано
- `skipped` — сколько сообщений пропущено

### `GET /api/v1/conversations`

Список собеседников.

Query params:
- `sort`: `last_message_at` | `message_count`
- `order`: `asc` | `desc`
- `limit`
- `offset`

Response:

```json
{
  "items": [
    {
      "user_id": 4043901,
      "display_name": "Имя из архива",
      "message_count": 15234,
      "last_message_at": "2021-08-30T19:21:38Z",
      "first_message_at": "2012-03-01T10:15:00Z"
    }
  ],
  "total": 1234
}
```

### `GET /api/v1/messages/{user_id}`

Получение окна сообщений переписки.

Query params:
- `limit`
- `before`
- `after`
- `around`

Правила:
- `around` нельзя комбинировать с `before` и `after`;
- `around` используется для прыжка к сообщению из поиска;
- пагинация строится по timestamp, не по большим offset.

### `POST /api/v1/search`

Полнотекстовый поиск по сообщениям.

Request:

```json
{
  "query": "сумерки",
  "user_id": 4043901,
  "date_from": "2020-01-01T00:00:00Z",
  "date_to": "2020-12-31T23:59:59Z",
  "limit": 50,
  "offset": 0,
  "mode": "simple"
}
```

### `POST /api/v1/export`

Экспорт сообщений.

Request:

```json
{
  "format": "jsonl",
  "user_id": 4043901,
  "date_from": "2020-01-01T00:00:00Z",
  "date_to": "2020-12-31T23:59:59Z",
  "limit": 1000
}
```

Форматы:
- `txt`
- `json`
- `jsonl`

## Frontend

### Основные страницы

- `Переписки`
- `Поиск`
- `Экспорт`
- `Импорт`

### Требования

- верхнее меню — маршруты;
- если `/api/v1/conversations` возвращает пустой список, UI сразу предлагает импорт;
- поиск работает по тексту сообщений;
- переход из поиска в переписку делается через `user_id + timestamp`;
- экран переписки поддерживает пагинацию и бесконечный скроллинг;
- frontend должен явно показывать, что включён расширенный режим поиска `fts`.

## Экспорт

Экспорт:
- может быть по одному пользователю или по всем сообщениям;
- поддерживает фильтры по времени;
- по умолчанию идёт в хронологическом порядке.
